# Throttler

## [basic::Throttler](include/basic/throttler.hpp) Блокирующий вариант без поддержки порядка выполнения поступивших запросов (не FIFO). Использует сервисный поток для добавления токенов.
Реализует алгоритм "leaky bucket" для контроля частоты выполнения некоторой секции кода.
Позволяет запрашивать разрешение на выполнение посредством вызова блокирующего вызова `Throttle()`, 
который или гарантирует немедленный доступ(хотя бы 1 токен доступен) или блокирует вызывающий поток до появления свободного токена.

Вызывающий метод `Throttle()` поток в случае отсутствия свободных токенов ожидает на условной переменой `cv_`, для работы условной переменной представлен мьютекс `mtx_`. Также мьютекс служит
для защиты разделяемой переменной `currentTokensNum_`, которая представляет текущее количество
доступных токенов. Каждый вызов `Throttle()` уменьшает количество доступных токенов на 1.

В случае, если поток- клиентa проснулся на вызове `Throttle()` в момент вызова деструктора объекта throttler, выбрасывается исключение.

Для защиты от состояния гонки которая может возникнуть, когда деструктор throttler'a уже выполнился, но есть активные потоки, которые могут находиться внутри вызова `Throttle()`, вводится переменная `activeThrottleCalls_` (управляется по RAII внутри вызова `Throttle()`). Деструктор throttler'a ждет пока
потоки не покинут `Throttle()`.

Для добавления токенов используется служебный поток `addTokenThread_`.
Частота добавления токенов зависит от параметра конструктора `tokensPerSecond`.
Добавляющий токены поток засыпает на время, расчитанное исходя из значения `tokensPerSecond`.
Максимальное количество доступных токенов задается параметром конструктора `maxTokens`.

## Преимущества
- простота реализации;
- нет накладных расходов памяти;

## Недостатки
- создается дополнительный поток, потребляющий ресурсы;
- нет поддержки порядка выполнения поступивших запросов;
- ожидающий токена поток блокируется;

## [no_service_thread::Throttler](include/no_service_thread/throttler.hpp) Блокирующий вариант без поддержки порядка выполнения поступивших запросов. Вариант без отдельного служебного потока.

Служебный поток не создается.
Добавление токенов происходит в потоке- клиенте по
наступлению достаточного для добавления времени. Время последенго добавления токена запоминается, на основе этого значения происходит расчет количества токенов, которые можно добавить.

## Преимущества
- простота реализации;
- нет накладных расходов памяти;
- нет дополнительного сервисного потока;

## Недостатки
- нет поддержки порядка выполнения поступивших запросов;
- поток ожидающий токен блокируется;
